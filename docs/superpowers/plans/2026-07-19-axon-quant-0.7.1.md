## Context

0.7.0 release notes 明确记录了 API 缺陷
(`docs/superpowers/plans/2026-07-18-axon-quant-0.7.0.md:96`):

> **🐛 已知 API 缺陷**:`begin_bar_multi(legs: dict[instrument, price])` 不可用
> —— dict 不可哈希作为 dict key,Python 端无法构造 `dict[instrument_dict, price]`。
> workaround:连续调 2 次 `begin_bar`(per-bar 末次 rebalance + funding 调度各 1 次)。
> **0.7.1 应改 `begin_bar_multi` 接受 `list[tuple[instrument, price]]`**(最干净),
> 或让 `parse_instrument` 支持 tuple key

PR-A 落地"list[tuple]"方案,理由:

- `list[tuple]` 是 Python 标准 dict-like 结构,`[(k1, v1), (k2, v2)]` 与 dict 同语义
- 实现简单,只改 `begin_bar_multi` 一个函数签名 + 循环
- 不破坏现有 0.7.0 用户的 workaround(连续调 2 次 `begin_bar` 仍工作)
- 与 `with_seed_liquidity_for` 已有的 per-leg 注入风格一致

PR-B / PR-C / PR-D 是 0.7.0 review 与 quantcell 集成过程中暴露的痛点,合并到 0.7.1 release 以保证「一次升级拿到全部修复 + 体验改进」。

## Task 1:改 `begin_bar_multi` 签名

**File:** `crates/axon-backtest/src/python/engine.rs:280-289`

### Step 1.1:替换 `&Bound<PyDict>` 为 `&Bound<PyList>`

定位 `crates/axon-backtest/src/python/engine.rs:280`:

```rust
fn begin_bar_multi(&mut self, _py: Python<'_>, legs: &Bound<'_, PyDict>) -> PyResult<()> {
    let mut parsed: Vec<(axon_core::types::Instrument, f64)> = Vec::with_capacity(legs.len());
    for (key, value) in legs.iter() {
        let inst = super::types::parse_instrument(key.cast::<PyDict>()?)?;
        let price: f64 = value.extract()?;
        parsed.push((inst, price));
    }
    self.inner.begin_bar_multi(parsed);
    Ok(())
}
```

改为:

```rust
/// 0.7.0 新增,0.7.1 修正:多 leg 同 bar seed(spot + perp 套利场景)
///
/// 接受 `list[tuple[instrument_dict, price]]` 而非 `dict`(0.7.0 dict 形式
/// 因 dict 不可哈希不可用,见 2026-07-18 plan §Phase 5.6 已知缺陷)。
///
/// 在同一根 bar 内对多个 instrument 同时 seed liquidity,常用于 delta-neutral
/// 套利(spot 和 perp 各自挂对手盘,策略单两边都能成交)。
///
/// Args:
/// - `legs`: `list[tuple[instrument_dict, price]]`,
///   第一项是 instrument dict(由 `spot_instrument()` / `swap_instrument()` 工厂构造),
///   第二项是该 leg 的 mid_price
///
/// Example:
/// ```python
/// engine.begin_bar_multi([
///     (spot_instrument("BTC", "USDT"), 100.0),
///     (swap_instrument("BTC", "USDT", "UsdMargin", 1.0), 200.5),
/// ])
/// ```
///
/// 与多次 `begin_bar(price, instrument)` 的区别:
/// - 多次 `begin_bar` 会 bar_id 自增多次 + funding 调度多次 + 末次 rebalance,
///   **不**适合多 leg 同 bar
/// - `begin_bar_multi` 调一次,bar_id +1,funding 调度一次,末次 rebalance
#[pyo3(signature = (legs))]
fn begin_bar_multi(&mut self, legs: &Bound<'_, PyList>) -> PyResult<()> {
    let mut parsed: Vec<(axon_core::types::Instrument, f64)> = Vec::with_capacity(legs.len());
    for item in legs.iter() {
        let tuple: &Bound<'_, PyTuple> = item.cast()?;
        if tuple.len() != 2 {
            return Err(PyValueError::new_err(format!(
                "begin_bar_multi leg must be (instrument_dict, price) tuple, got len={}",
                tuple.len()
            )));
        }
        let inst = super::types::parse_instrument(tuple.get_item(0)?.cast::<PyDict>()?)?;
        let price: f64 = tuple.get_item(1)?.extract()?;
        parsed.push((inst, price));
    }
    self.inner.begin_bar_multi(parsed);
    Ok(())
}
```

**注**:需在文件顶部 `use pyo3::types::{PyDict, PyList, PyTuple};` 加 `PyTuple`。

## PR-B 实施:`bar_nav_curve` per-bar 估值曲线(2026-07-19)

## 背景

0.7.0 的 `RunResult.equity_curve` 只在 fill / mark / funding 事件触发时采样,
**无事件 bar 不留帧**。后果:

- 短回测 + 全无 fill → 末帧 = `initial_cash`,得不出 NAV 波动率
- `sharpe_ratio` 沿用 `equity_curve` 算 log return → 退化为 0
- `max_drawdown` 同理失真(无 NAV 波动即无回撤)

quantcell baseline 服务 `BaselineBacktestService` 报 sharpe_ratio=0.0 的根因。

## 方案

新增 `RunResult.bar_nav_curve: Vec<(Timestamp, f64)>`,在 `begin_bar` /
`begin_bar_multi` 收尾时**每 bar 一帧**采样,时间戳 = `clock.now()`,
NAV = `compute_nav(ts, mid_price)`(用 `mark_cache` + `mid_price` fallback)。

## 关键代码

- `crates/axon-backtest/src/engine.rs`:
  - `BacktestState` 新增 `bar_nav_curve: Vec<(Timestamp, f64)>`(`#[derive(Default)]` 自动 init)
  - `RunResult` 新增 `pub bar_nav_curve: Vec<(Timestamp, f64)>` + `Default` 段 init
  - `sample_bar_nav(mark_fallback: f64)` 辅助函数:
    - `ts = clock.now()`,`nav = compute_nav(ts, mark_fallback)`
    - de-dup:同 ts 覆盖末帧(同 bar 多次 `begin_bar` 罕见但可能)
  - `begin_bar` / `begin_bar_multi` 末尾各调一次
  - `build_result` 拷到 `RunResult.bar_nav_curve`
- `crates/axon-backtest/src/python/engine.rs`:
  - `PyRunResult::bar_nav_curve` getter(同 `equity_curve` 模式,`list[tuple[i64, f64]]`)
  - `to_dict()` 加 `bar_nav_curve_points` 计数

## PR-C 实施:`with_*` 统一返回 `&mut Self` 支持链式(2026-07-19)

## 背景

0.7.0 的 `BacktestEngine.with_*` 方法(如 `with_fee_config` /
`with_auto_rebalance` / `with_seed_liquidity` / `with_funding_schedule` 等)
存在两类不一致的返回值:

- **部分返回 `()`**:Rust 端 `pub fn with_fee_config(&mut self, rate: f64)`,
  Python 绑定 `fn with_fee_config(&mut self, rate: f64)`
- **部分返回 `&mut Self`**:`with_force_liquidate`、`with_funding_schedule_disable`

后果:

- Python 端无法 `engine.with_x().with_y().with_z()` 链式,得写多行重复
- 调用方靠记忆判断"这个 `with_*` 返回啥",心智负担
- 与 `set_target_position(...)` / `begin_bar_multi(...)` 风格不统一

## 方案

统一所有 `BacktestEngine::with_*` 方法返回 `&mut Self`(Rust)/
`PyRefMut<'py, Self>`(Python),支持 `engine.with_x().with_y()` 链式。

## 实施范围

| 方法 | Rust 端返回 | Python 端返回 |
|------|-------------|---------------|
| `with_matching_engine` | `&mut Self` | `PyRefMut<Self>` |
| `with_fee_config` | `&mut Self` | `PyRefMut<Self>` |
| `with_force_liquidate` | `&mut Self`(原本就 chainable) | `PyRefMut<Self>` |
| `with_seed_liquidity` | `&mut Self` | `PyRefMut<Self>` |
| `with_seed_liquidity_for` | `&mut Self` | `PyRefMut<Self>` |
| `with_auto_rebalance` | `&mut Self` | `PyRefMut<Self>` |
| `with_auto_rebalance_disable` | `&mut Self` | `PyRefMut<Self>` |
| `with_funding_schedule` | `&mut Self` | `PyRefMut<Self>` |
| `with_funding_schedule_disable` | `&mut Self`(原本就 chainable) | `PyRefMut<Self>` |

## 关键代码

`crates/axon-backtest/src/python/engine.rs`:

```rust

    threshold: f64,

```

## PR-D 实施:`sharpe_ratio` 样本不足警告 + bar 间隔归一化(2026-07-19)

## 背景

0.7.0 的 `TradingMetrics::sharpe_ratio(periods_per_year: f64)` 存在两个痛点:

1. **样本不足静默返回 0**:当 `log_return_count < 2` 时直接 `return 0.0`,
   没有 tracing 警告。短回测 / 单调行情下用户得不到反馈,误以为公式算错
2. **bar 间隔年化因子必须手算**:调用方需自己算
   `periods_per_year = (365 * 24 * 3600) / bar_interval_secs`,
   然后再 `sqrt(periods_per_year)`。**当前 `BacktestEngine::build_result` 还
   错传了 `35_040_f64.sqrt()`**(`35_040` = 15-min bars per year),内部又
   `* periods_per_year.sqrt()`,实际得到 `sqrt(sqrt(35040)) ≈ 13.7`,
   比正确年化因子 `sqrt(35040) ≈ 187.2` 小一个数量级

quantcell 看到 `sharpe_ratio` 极小是这两个 bug 叠加的结果。

## 方案

`axon_core::metrics::TradingMetrics` 新增两个方法:

```rust
/// 直接传 periods_per_year(原方法,doc 加强)
pub fn sharpe_ratio(&self, periods_per_year: f64) -> f64;

/// 0.7.1 新增:传 bar 持续秒数,自动算 periods_per_year = 365*24*3600 / secs
pub fn sharpe_ratio_annualized(&self, bar_duration_secs: f64) -> f64;
```

- 样本数 < 30 时 `tracing::warn!` 提示统计意义不足
- `sharpe_ratio_annualized` 内部 `periods_per_year.sqrt() * mean / std`,
  调用方不再需要手算

## 关键代码

`crates/axon-core/src/metrics/trading_metrics.rs`:

```rust
/// 便捷方法:传 bar 持续秒数,自动算 periods_per_year = 365*24*3600/secs
///
/// 比手算 `sharpe_ratio(periods_per_year)` 更安全(避免漏乘 sqrt)。
///
/// Example:
/// - 15-min bar → 900.0
/// - 1h bar → 3600.0
/// - 1d bar → 86_400.0
pub fn sharpe_ratio_annualized(&self, bar_duration_secs: f64) -> f64 {
    const SECS_PER_YEAR: f64 = 365.0 * 24.0 * 3600.0;
    let periods_per_year = SECS_PER_YEAR / bar_duration_secs;
    self.sharpe_ratio(periods_per_year)
}
```
