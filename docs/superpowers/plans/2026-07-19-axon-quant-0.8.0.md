#### 1.1 Scheduler lifetime 重构(独立 PR `refactor(scheduler): replace *mut EventQueue with safe abstraction`)

**Root cause**:`crates/axon-core/src/scheduler/{core,context}.rs` 用 `*mut EventQueue`
绕过 Rust 借用检查(避免 `Rc<RefCell>` 性能损耗),5 处 unsafe 路径需要重新审计。

**方案**:

- 方案 A:`Arc<Mutex<EventQueue>>` — 简单安全,微秒级损耗
- 方案 B:`crossbeam::queue::SegQueue` — lock-free,无 unsafe
- 方案 C:`Rc<RefCell<EventQueue>>` — 单线程,borrow checker 友好
- **推荐 A**(可观测性 + 安全 + 微秒级损耗可接受)

**验收**:

- [ ] `crates/axon-core/src/scheduler/` 内 0 处 `unsafe`(除非 SAFETY 注释明确)
- [ ] `cargo clippy --workspace --all-targets -- -D warnings` 干净
- [ ] 现有 scheduler 单测 + e2e 全过
- [ ] 性能对比:`Arc<Mutex>` vs `*mut` 在 10K 事件压测下 < 5% 慢

### Phase 2:风险与组合深化(预计 4-5 周)

**目标**:把 0.7.0 Phase 4 minimal 补到生产可用

#### 2.1 B1 gamma / vega 接入真实 IV 源

**Root cause**:0.7.0 gamma/vega 全 0,因为无 mark 历史(`push_mark` 只存最新 1 帧)且无 IV 源(IV 来自期权链或 vol surface)。

**方案**:

- 新增 `MarketDataSource` trait(`axon-core::data` 模块)

  ```rust
  pub trait MarketDataSource: Send + Sync {
      fn mark_history(&self, instrument: &Instrument, lookback: usize) -> Vec<(Timestamp, f64)>;
      fn implied_vol(&self, instrument: &Instrument, strike: f64, tenor: f64) -> Option<f64>;
      // ...更多方法
  }
  ```

- 内置实现:
  - `InMemoryMarketData` — 单元测试用
  - `CsvMarketData` — 从 CSV 加载历史
  - (后续 Phase:`AkkaMarketData` / `BybitMarketData` 等)
- `PortfolioRiskEngine`:
  - `gamma_exposure()`:用 mark 历史估算 underlying 移动 ±1% 的 PnL 变化
  - `vega_exposure()`:`Σ (position × IV_sensitivity × contract_size)`
- `RunResult.risk_metrics`:
  - `portfolio_delta` 已有(0.7.0)→ 接入 contract_size(B2)
  - 新增 `portfolio_gamma` / `portfolio_vega` / `per_instrument_gamma` / `per_instrument_vega`

**验收**:

- [ ] `MarketDataSource` trait + 2 个内置实现
- [ ] gamma / vega 端到端测试(SPX 跨 instrument 实测 portfolio_gamma ≈ Σ instrument_gamma)
- [ ] 文档:中英版 `risk_metrics.md` 含 IV 源接入示例

#### 2.2 B2 per-instrument delta 接入 `contract_size`

- [ ] `SwapInstrument.contract_size` 字段已存在,`PortfolioRiskEngine` 计算时改为:

  ```rust
  delta_per_leg = position_qty * contract_size;  // 0.7.0 假设 unit_delta=1.0
  ```

- [ ] 测试:contract_size=0.01 的 ETH perp + spot BTC 组合,portfolio_delta 量级正确
- [ ] CHANGELOG 0.8.0 段说明:delta 语义与之前可能不同(0.7.0 隐式 1.0)

#### 2.3 B3 跨 instrument gamma 协方差

- [ ] `PortfolioRiskEngine::gamma_covariance_matrix() -> HashMap<(Instrument, Instrument), f64>`
- [ ] 数值稳定性:用 Cholesky 分解或 Ledoit-Wolf 收缩
- [ ] 测试:2 instrument 组合 portfolio_gamma = leg1_gamma + leg2_gamma + 2 * cov

#### 2.4 B4 `equity_curve` vs `bar_nav_curve` 整合

**现状**:

- `equity_curve`:fill / mark / funding 事件触发时采样(末帧 mark 估值)
- `bar_nav_curve`:每 bar 一帧(短回测也得波动率)

**决策**:

- 方案 X:**保留双曲线**,文档化各自适用场景(0.7.1 PR-B 已隐式采用)
- 方案 Y:**合并为单曲线**,`bar_nav_curve` 为主,`equity_curve` 退化为 fill-only 序列
- 方案 Z:**二选一**,以 `bar_nav_curve` 为主,删 `equity_curve`(BREAKING)
- **推荐 X**(风险最低,语义最清晰)

**实施**:

- [ ] `RunResult` 文档:中文 doc 表格对比两个字段的"采样时机 / 用途 / 计算量"
- [ ] `TradingMetrics` 计算优先用 `bar_nav_curve`(更准)
- [ ] `equity_curve` 保留兼容 + deprecated 警告(0.8.0 仍可用,0.9.0 删除)

#### 2.5 B5 新风险指标

- [ ] `TradingMetrics::sortino_ratio(periods_per_year)` — 只算下行波动
- [ ] `TradingMetrics::calmar_ratio(periods_per_year)` — 年化收益 / 最大回撤
- [ ] `TradingMetrics::information_ratio(benchmark, periods_per_year)` — 超额收益 / tracking error
- [ ] 统一样本不足警告:抽 `assert_sample_size(n: usize, threshold: usize, metric: &str) -> Result<(), Warn>`

### Phase 3:撮合层重写(预计 4-5 周)

**目标**:L3 matching engine 完整重写,性能 gate < 2x vs L2

#### 3.1 A1 L3MatchingEngine 完整重写

**0.7.0 现状**:L3Book 只是 L2 引擎的"稳定只读视图",撮合本体还是 L2

**0.8.0 方案**:

- 新增 `L3MatchingEngine`(独立 struct,不复用 L2)
  - `BTreeMap<Price, Vec<L3Order>>` bids + asks
  - `submit(order)`:price-time priority,partial fill 跟踪到 `L3Order.filled_qty`
  - `cancel(order_id)`:从 book 移除
  - `best_bid()` / `best_ask()` / `depth(levels)`
- `L3Order { order_id, side, qty, filled_qty, timestamp_ns, source }`
- `L3Fill` 跟踪每笔 partial fill 的"被吃方"
- `L3Book::from_l3_engine()` 工厂

**性能 gate**:

- [ ] bench 脚本:`benches/l3_vs_l2.rs` 同 case 对比
- [ ] gate:L3 撮合 ≤ 2x L2 时间 + ≤ 2x 内存
- [ ] 不达 gate → 降级为 `l3-matching` feature flag,默认 L2

#### 3.2 A2 撮合层多态扩展

- [ ] `Strategy` trait:

  ```rust
  pub trait Strategy: Send {
      fn on_fill(&mut self, fill: &FillRecord, ctx: &mut StrategyContext) -> Vec<Order>;
      fn on_bar(&mut self, bar: &Bar, ctx: &mut StrategyContext) -> Vec<Order>;
  }
  ```

- [ ] `MultiAssetStrategy` trait:多 leg 协同决策
- [ ] `EngineRouter`:根据 `Strategy` 路由到 `Box<dyn MatchingEngine>`
- [ ] 测试:`Box<dyn Strategy>` + `Box<dyn MatchingEngine>` 联动

#### 3.3 A3 撮合 hot-path 性能优化

**0.7.0 bench 测得**:`inner.submit` ~150µs 占主导,`compute_impact` ~100ns

**优化方向**:

- [ ] **SIMD 价格比较**:`f64` 价格排序用 `packed_simd` 或 `std::simd`(MSRV 检查)
- [ ] **Arena allocator**:每 bar 内的 fill 记录用 `Bump` allocator 一次性回收
- [ ] **订单结构紧凑**:`Order` 从 ~256B 压到 ~64B(`#[repr(C, packed)]` 谨慎使用)
- [ ] **SoA vs AoS**:`L1Book` 内部 bids/asks/timestamps 改 SoA
- [ ] bench 对比:优化前 vs 优化后,目标 `inner.submit` ≤ 50µs

## Risks / Open Questions

### 高风险

1. **A1 性能 gate 可能不达** — L3 完整重写若 > 2x L2 慢,降级为 feature flag。`l3-matching` 默认关,0.9.0 重新评估

1. **B3 协方差数值稳定** — 高维协方差矩阵可能 singular,需 Ledoit-Wolf 收缩
2. **B4 双曲线决策** — 用户决定 X/Y/Z 中哪个,影响 0.9.0 API
3. **D1 RL 训练稳定性** — PPO 收敛对超参敏感,需 ray tune HPO 兜底
4. **D2 docs 完整同步** — 双倍工作量,Phase 5 收尾可能漏

2. **D1 HPO 后端**:`ray[tune]` / `optuna` / 自研?
