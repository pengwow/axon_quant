> 在 0.6.0 范围内,完成 0.5.0 known limitations 全部收口,实现多 leg 抽象
> 从"backtest 路径"扩展到"全栈统一",并把 Phase C/D 留下的手动 hook
> 自动化。

## 范围(0.6.0 强制收口)

按 0.5.0 `CHANGELOG.md` 中 Known Limitations (0.6.0 Roadmap) 列出 6 项,本次全部收口:

|------|------|--------|--------|

每个 Phase 单独一个 commit,每 commit 跑 `make test-ci` 验证行为不变。

## 设计决策(已与用户对齐)

| 决策点 | 选择 | 理由 |
|--------|------|------|
| OMS 架构 | **渐进**:加 `instrument` 字段,保持 Decimal 并行 | 与 `axon_core::Position(symbol + instrument 双保留)模式一致`;不破坏 OMS 作为 live trading 独立路径的定位 |
| streaming BREAKING | **全面 BREAK**:trait + enum 一刀切 | 0.x 项目风格,一次抓大重写比双轨更干净;0.5.0 已经做了一次 BREAKING,用户已接受 |

## Phase 1:`begin_bar` 收尾自动 rebalance

### 触发点
`begin_bar(mid_price, instrument)` 收尾追加 `self.rebalance_to_target(None)` 调用。
用 `None` 走 `auto_rebalance_threshold` 配置(`with_auto_rebalance(threshold)` 启用)。

### 关键 API
- **复用既有** `rebalance_to_target(None)`(0.5.0 Phase D 已实现,无新代码)
- **加 guard**:`bar_id: u64` 计数器;每次 `begin_bar` 自增,`rebalance_to_target` 检查"本 bar 已 rebalance"则直接 return
  - 同根 bar 多次调 `begin_bar`(实际场景罕见,但代码允许)只触发一次 rebalance

### 风险与边界
- 策略层手写 `rebalance_to_target` 与自动模式同时启用:行为叠加 → 在 doc 明确"自动模式下手动调用 = no-op"(guard 天然保证)
- `with_auto_rebalance_disable()` 后 `begin_bar` 不调 rebalance(同现状)

### 测试
- `crates/axon-backtest/src/engine.rs::tests` 新增:`begin_bar_auto_rebalance_triggers_per_bar` /
  `begin_bar_auto_rebalance_disable_noop` / `bar_id_guard_prevents_double_rebalance`
- `crates/axon-integration-tests/src/delta_neutral_arb.rs`:把现有 3 个 `rebalance_*` 测试的"手动 rebalance_to_target"删掉,改为依赖 `begin_bar` + `with_auto_rebalance`

## Phase 2:Funding 8h 自动调度

### 配置
```rust
pub struct FundingSchedule {
    pub instrument: Instrument,
    pub interval_ns: i64,       // 典型 8h = 8 * 3600 * 1_000_000_000
    pub fixed_rate: f64,        // 资金费率(正=long 付,负=long 收)
    pub mark_aware: bool,       // true:用 mark_cache;false:用 fallback mark
}
```

### 关键 API
- `BacktestEngine::with_funding_schedule(schedule) -> &mut Self`
- `BacktestEngine::with_funding_schedule_disable(instrument) -> &mut Self`
- `BacktestEngine::funding_schedules() -> &HashMap<Instrument, FundingSchedule>`(读)

### 调度钩子
`begin_bar` 收尾(在 Phase 1 rebalance 之前)遍历 active schedules:
- `next_funding_ts(prev_funding_ts) = prev_funding_ts + interval_ns`
- 当 `next_funding_ts <= bar_ts` → 合成 `FundingEvent { instrument, rate: fixed_rate, mark: mark_cache[instrument] 或 fallback, ts: next_funding_ts }` 推 queue
- **while 循环**:bar 跨多个 8h 边界(回测跳秒场景)→ 多次推 funding 直到 `next_funding_ts > bar_ts`
- 维持 `last_funding_ts: HashMap<Instrument, Timestamp>` 状态

### 风险与边界
- 与现有 `push_funding(instrument, rate, mark, ts)` 共存:用户手动 push 的 funding 走 `Event::Funding` 路径,与自动 push 完全一致,无冲突
- 取消调度(清空 schedule)不影响历史 funding PnL 累计

### 测试
- `crates/axon-backtest/src/engine.rs::tests` 新增:`funding_schedule_basic_8h` /
  `funding_schedule_skip_8h_boundary_crossed` / `funding_schedule_disabled_no_settlement` /
  `funding_schedule_mark_aware_uses_cache`
- `crates/axon-integration-tests/src/delta_neutral_arb.rs`:`auto_funding_delta_neutral_8h_schedule`

## Phase 6:跨 leg 风险约束

### 新类型
`crates/axon-core/src/types/pair.rs`(新):
```rust
#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub struct LegPair {
    pub spot: Instrument,
    pub perp: Instrument,
    pub hedge_ratio: f64,  // 通常 1.0(同标的 spot+perp 1:1 对冲)
}
```
`axon_backtest::matching::l3::CrossPair` 改为 `LegPair` 薄包装(内部存 `LegPair` 字段,`max_quantity` 等 L3-specific 字段保留)。

### 新风险检查
`crates/axon-risk/src/checks/` 新增:
- **`pair_net_exposure.rs`**:对每个 `LegPair`,`net_delta = qty(spot) - qty(perp) * hedge_ratio`;`|net_delta| > config.max_pair_drift` → 拒 rebalance 单
- **`per_leg_var.rs`**:把 `pnl_history: VecDeque<f64>` 扩为 `HashMap<Instrument, VecDeque<f64>>`;`update_daily_pnl(pnl: f64, instrument)` 改 signature;`calculate_var` 加 per-instrument 版本
- **`stress_test.rs`**:对每 leg 应用 shock(`StressScenario { instrument, price_shock_pct, rate_shock_bps }`),算 hypothetical NAV

### 新 API
- `RiskConfig` 加字段:`max_pair_drift: f64`(默认 0.0=禁用)/ `leg_pair_registry: HashMap<String, LegPair>` / `stress_scenarios: Vec<StressScenario>`
- `RiskEngine::check_portfolio_pair(portfolio, pair_registry) -> Vec<RiskAlert>`
- `RiskEngine::stress_test(portfolio, scenarios) -> StressTestResult`
- `RiskEngine::per_leg_var(returns_per_leg: &HashMap<Instrument, &[f64]>, confidence: f64) -> HashMap<Instrument, f64>`

### BREAKING
- `update_daily_pnl(pnl: f64)` → `update_daily_pnl(pnl: f64, instrument: Instrument)` —— 任何外部 risk consumer 需重写
- `pnl_history` 字段类型从 `VecDeque<f64>` 变 `HashMap<Instrument, VecDeque<f64>>` —— `RiskEngine` trait 字段访问需更新

### 测试
- `crates/axon-risk/src/checks/pair_net_exposure.rs::tests` 新增 ~6 个
- `crates/axon-risk/src/checks/per_leg_var.rs::tests` 新增 ~4 个
- `crates/axon-risk/src/checks/stress_test.rs::tests` 新增 ~4 个
- `crates/axon-integration-tests/src/contract.rs` 风险契约测试扩展

### 风险与边界
- `axon-risk` 当前不依赖 `axon-backtest`;引入 `LegPair` 后仍不直接依赖(`LegPair` 在 `axon-core`),`CrossPair` 的 L3 特定字段不污染 risk 层
- 跨 leg VaR 的 covariance matrix 计算复杂度:`HashMap<Instrument, &[f64]>` 输入,O(n²) 协方差估算;n=leg 数通常 2~10,无性能问题

## 整体顺序与依赖图

```
P1 (begin_bar auto-rebalance) ─┐
                               ├─→ P3 (L3) ─→ P4 (streaming) ─→ P5 (OMS) ─→ P6 (cross-leg risk)
P2 (funding schedule) ─────────┘                  │                │              │
                                                └─ 都依赖 axon_core::Instrument(0.5.0 已完成) ─┘
                                                └─ P6 额外依赖 axon_core::LegPair(本次 P3 同时新增)
```

依赖关系细化:
- P1 不依赖任何 Phase
- P2 不依赖任何 Phase
- P3 必须先有 P1+P2(因 `begin_bar` 收尾会调 rebalance + funding schedule,L3 内 `run_auction` 不直接依赖,但测试时序上要 P1+P2 已合)
- P4 依赖 P3(streaming 内 L1 + L3 routing 模式要同步;CSV 从 symbol 改 instrument 同根)
- P5 依赖 P3+P4(OMS snapshot 与 streaming snapshot 都要走 Instrument 序列化,需保持一致)
- P6 依赖 P5(per-leg VaR 要看 Portfolio 的 Instrument 化是否完成)

## 决策记录

1. **OMS 不合并** —— Decimal 精度是 live trading 必要属性;`axon-core` i64×1e6 与 `axon-oms` Decimal 是两个有意分离的精度体系
2. **streaming 一刀切 BREAKING** —— 0.x 项目风格,0.5.0 `Order::new → spot/swap` 已开过 BREAKING 前例,用户接受
