## Per-bench observations

### `l1_submit` (baseline) ‚Äî 1.37¬µs median

Pure L1 price-time priority. No L2/L3 wrappers. This is the floor for any
matching operation in the crate.

### `l2_submit` ‚Äî 1.38¬µs median (L1 + 0.3%)

L2 adds `order_index` HashMap + `stats` counters. Effectively free.

### `l3_single_asset_submit` ‚Äî 1.47¬µs median (L1 + 7.2%)

L3 routing + PartialFillTracker (per-fill FillRecord::new √ó 2 + HashMap
entry √ó 2 + is_filled √ó 1). The 7% overhead is the cost of full per-fill
metadata tracking introduced in A1.1.

### `l3_multi_asset_submit` ‚Äî 1.59¬µs median (L1 + 15.3%)

Same as single asset but with 5-instrument rotation. The extra ~110ns comes
from HashMap<Instrument, L2Engine> lookup on each submit (the
`engines.get(&order.instrument)` call). Still well below 2x gate.

### `l3_depth_scaling/10/50/100/500` ‚Äî 1.43 / 1.50 / 1.65 / 1.62¬µs

BTreeMap-based price ladder is O(log n) as expected. 50x depth growth
(10 ‚Üí 500) yields only 14% latency increase. The 100-depth outlier in
criterion's `change:` line (+63% vs stored base) reflects an unusually
high-variance run; absolute median 1.65¬µs is still within normal range.

| L1 submit        | 1.04¬µs | 1.37¬µs | +32% |

The A1.1 numbers came from `--quick` mode (1 sample, fewer iterations).
The A1.3 numbers are statistically more reliable. The +20-40% delta is
not a real regression ‚Äî it's the difference between --quick and default
criterion config catching real variance.
